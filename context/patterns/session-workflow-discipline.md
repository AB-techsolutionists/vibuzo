# Session Workflow Discipline

The correct workflow rhythm for session management to prevent information loss and file overlap.

## Golden Workflow

```
/session → /new
```

In detail:
1. **`/session`** — Generate a comprehensive summary before any context-truncating operation. This creates the `context/sessions/YYYY-MM-DD-<title>.md` file with every user request, tool call, file change, decision, and git state. The Session Compaction block is auto-generated.
2. **`/new`** — Start a fresh session. Open the session file, copy the Session Compaction block (the full styled box), and paste it as starting context. The auto-load chain (`context/index.md` → `sessions/index.md` → latest session file → Session Compaction section) picks it up automatically.

## Trigger Points

Run `/session` (and optionally `/new`) at every natural breakpoint:

| Trigger | Why |
|---------|-----|
| **After finishing a feature** | Capture decisions, file changes, state before moving on |
| **Before switching tasks** | Preserve context for the unfinished task |
| **When conversation feels long** | ~15-30 messages is a good threshold before compaction is needed |
| **End of day** | Hard checkpoint — never end a day without a session file |
| **Always before `/compact`** | This is the hard rule — never lose context to compaction |

## Anti-Patterns

| Anti-pattern | Why it fails |
|--------------|--------------|
| **`/compact` then `/session`** | After compaction, the conversation history is truncated. `/session` can only see partial context — the narrative will be incomplete. |
| **Multiple `/session` without `/compact` between** | Creates overlapping session files that cover the same period. Leads to confusion about which file is authoritative. |
| **Skipping `/session` entirely** | No persistent record. All decisions, file changes, and context are lost on `/compact` or `/new`. |
| **Running `/session` from a subtask** | Subtask file writes don't persist to the workspace. The session file may be lost or empty. Context/session commands must run in the main agent session. |

## Session Compaction Section

The `## Session Compaction` section at the bottom of each session file is **auto-generated by `/session`** in a styled box format containing Goal, Constraints & Preferences, Progress (Done/In Progress/Blocked), Key Decisions, Next Steps, Critical Context, and Relevant Files. No manual `/compact` → paste step is needed.

## Key Rules

1. **Never `/compact` without `/session` first** — this is the single most important rule
2. **Never run `/session` twice in a row** without a `/compact` between them
3. **The Session Compaction section is auto-generated** — never manually edit it
4. **Context/session commands run in the main agent session** — never as subtasks (`subtask: true` must be absent from these commands)
5. **Timestamps for conversation events are approximate** — only git reflog and file modification times are reliable (see `context/standards/timestamp-reliability.md` if it exists)
