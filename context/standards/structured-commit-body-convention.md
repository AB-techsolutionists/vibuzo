# Structured Commit Body Convention

**Date:** 2026-06-07
**Status:** Active

## Rule

When generating a commit message (via `/commit` or manually), the commit body MUST contain a structured natural-language description of every change, written for developers to read smoothly and understand the intent.

## Format

```
<type:> <brief subject>

- <file modified>: <human-readable explanation of what changed and why>
- <file modified>: <human-readable explanation of what changed and why>

<Summary paragraph explaining the change holistically, in present tense,
imperative mood, for developers to understand the context and intent.>

Part of the `/commit` workflow.
```

## Examples

### Example 1 — Version bump + release notes

```
chore: bump 0.1.5 → 0.1.6

- Modified VERSION: bumped 0.1.5 → 0.1.6
- Updated versioning.md: added 0.1.6 entry documenting
  the hotfix

This bumps the version and updates release notes for the
hotfix. The patch increment reflects a backward-compatible
bug fix.

Part of the `/commit` workflow.
```

### Example 2 — Feature implementation

```
feat: add dark mode toggle

- Modified src/theme.js: added dark mode state management
  with system preference detection
- Updated src/components/Header.jsx: added toggle button UI
- Added src/styles/dark.css: created dark color scheme

This implements a dark mode toggle that respects system
preferences and allows manual override. The toggle is
persisted to localStorage.

Part of the `/commit` workflow.
```

## Guidelines

- **Every file modified** must be listed with a brief explanation of what changed and why
- **Summary paragraph** at the end explains the change holistically (not just what, but why)
- **Present tense, imperative mood** — "Bumps the version", not "Bumped the version" or "We bumped the version"
- **Natural language** — write for humans, not machines. Avoid jargon or overly terse bullet points
- **Footer line** `Part of the \`/commit\` workflow.` identifies the commit as generated by the workflow
