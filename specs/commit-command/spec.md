# Commit Command

*Specification — 2026-06-07*

## Principles

- **Safety-first**: Never push to GitHub — the user retains full control of what/when to push
- **Gate-protected**: Every mutation step requires explicit approval before execution
- **Transparent**: Show exactly what will be bumped, what the release notes say, and the full commit message before committing
- **Consistent**: Follow existing versioning conventions (semver from `versioning.md`) and git commit style
- **Self-contained**: The command lives as a single command file under `commands/` — no new dependencies

## Specification

### Overview

A new `/commit` command that automates the version bump → release notes update → git commit flow for Vibuzo itself. It reads the current state, asks the user what kind of bump (patch/minor), updates `VERSION` and `versioning.md`, builds a detailed commit message, commits, and then presents a report telling the user to push manually. Every step is gated behind approval gates.

### User Stories

- **As a Vibuzo maintainer**, I want to run `/commit` so I don't have to manually track which files to bump and what format the commit message should follow.
- **As a Vibuzo maintainer**, I want approval gates at each step so I can verify and cancel before anything is modified.
- **As a Vibuzo maintainer**, I want to see the exact commit message before it's committed so I can review its accuracy.
- **As a Vibuzo maintainer**, I want a final summary telling me the commit is ready and how to push it.

### Functional Requirements

1. **FR1 — Bump type selection**: Prompt the user for bump type: `(p)atch`, `(m)inor`, or `(c)ustom`. Default: patch.
2. **FR2 — Version calculation**: Read `VERSION` file, calculate new version based on bump type following `versioning.md` rules (patch: +1, minor: +1 to minor, patch reset to 0).
3. **FR3 — Approval gate**: Show the proposed bump (`0.x.x → 0.x.x`) and ask for approval before writing anything.
4. **FR4 — Bump VERSION**: Write the new semver to `VERSION` file.
5. **FR5 — Bump release notes**: Prepend a new entry to the "Current Version" section in `context/standards/versioning.md` with the new version, date, and description.
6. **FR6 — Release notes content**: Prompt the user for a brief description of what changed (or accept it from the `/commit` arguments). Default: auto-generated from context.
7. **FR7 — Commit type selection**: Prompt the user for conventional commit type from: `feat:`, `fix:`, `refactor:`, `chore:`, `docs:`, `style:`, `test:`, `perf:`. Auto-detect from arguments if prefixed (e.g., `/commit "feat: add dark mode"`). Default: `chore:`.
8. **FR8 — Build commit message**: Construct a conventional commit message with:
   - Subject line: `<type>: <description>` (type from FR7, description from user/args)
   - Body: structured natural-language description of every change, auto-generated by the agent:
     - List each file modified with a brief human-readable explanation of what changed and why
     - Include a summary paragraph at the end explaining the change holistically
     - Written in present tense, imperative mood, for developers to read smoothly
   - Footer: "Part of the `/commit` workflow."
9. **FR9 — Commit gate**: Show the full proposed commit message and ask for approval before committing.
10. **FR10 — Git commit**: Run `git add` on the changed files (`VERSION`, `versioning.md`), then `git commit -m "subject" -m "body"`.
11. **FR11 — No push**: Do NOT run `git push`. After commit completes, display a report box.
12. **FR12 — Report box**: Show a matching VIBUZO-style box (`╔═╗║╚═╝`, 59-char width) with:
    - Commit hash (abbreviated)
    - Files changed
    - New version
    - Push command: `git push`
    - Instructions telling the user to manually push

### Acceptance Criteria

| # | Criterion | FR |
|---|-----------|----|
| AC1 | Running `/commit` without arguments prompts for bump type with default patch | FR1 |
| AC2 | Running `/commit "fix: fixed login bug"` uses the argument as the change description | FR6 |
| AC3 | Running `/commit "feat: add dark mode"` auto-detects type `feat:` from argument prefix | FR7 |
| AC4 | Bump calculation respects versioning.md rules (patch 0→19, minor rollover) | FR2 |
| AC5 | A gate pauses before VERSION is written showing old → new version | FR3 |
| AC6 | VERSION file is updated to the new semver | FR4 |
| AC7 | A new entry is prepended to the "Current Version" section in versioning.md | FR5 |
| AC8 | The commit message follows `<type>: <description>` format and is gated for approval | FR8, FR9 |
| AC9 | `git push` is never called — not even as a dry-run | FR11 |
| AC10 | The report box includes the commit hash, new version, changed files, and push instruction | FR12 |
| AC11 | The command file follows the same YAML frontmatter + description pattern as other commands in `commands/` | — |

### Out of Scope

- Automatic push to GitHub (never — by design)
- Multiple commit workflows (single commit per invocation)
- Pull request creation
- Integration with CI/CD
- Changelog generation beyond versioning.md
- Multi-repo support
